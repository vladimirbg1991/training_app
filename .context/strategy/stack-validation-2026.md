# A 2026 fitness-app stack, validated and sharpened

**Your proposed stack is ~90% right and is the strongest mainstream choice in 2026 for a Hevy/Strong-tier fitness tracker.** The headline change: replace **Upstash Redis (pay-as-you-go) with Redis Cloud or Upstash Fixed** for BullMQ; move BullMQ workers off Supabase Edge Functions onto **Fly.io or Railway**; insert **Clerk** as the identity provider in front of Supabase (with Supabase Auth as your migration safety net once you cross ~200k MAU); commit to **RevenueCat now, layer Superwall later** for paywalls; and accept that the **Apple Watch app must be native SwiftUI**, not React Native. Everything else — RN+Expo, PowerSync, Supabase Postgres, Drizzle, Mux, NativeWind, FlashList, Reanimated, MMKV, Sentry — survives 2026 scrutiny intact.

The only stack-defining risk you carry is **Supabase auth-MAU pricing**, which becomes punishing past ~200k MAU ($3,250/mo per million MAU) and should drive a planned migration path to Clerk-issued JWTs from day one. Fitness apps that ship in 2026 without **Live Activities, passkeys, native Apple Watch, and an offline-first SQLite layer** will feel a generation behind Hevy/Strong/Fitbod — your stack covers all four.

A note on background research: Hevy's technical co-founder is **Desmond McNamee** (with co-founder Guillem Ros, both ex-8fit), not "Carlos Souto" as your brief suggested. The Hevy stack has been publicly confirmed as **React Native + Mobx + React Navigation + Victory Native** on the client and **Node.js + Express + PostgreSQL** on the server, with native Swift/Kotlin only for thin shims (likely Watch/Wear). Your proposed stack is, in effect, "Hevy plus a real local-first sync engine and PowerSync-grade conflict semantics."

## React Native and Expo are at peak maturity in 2026

**RN 0.82 (Oct 2025) permanently disabled the Old Architecture; 0.84 (Feb 2026) made Hermes V1 the default and shipped a 0.84 binary that's ~3 MB smaller and builds 20% faster on iOS.** By January 2026, ~83% of Expo SDK 54 EAS builds run the New Architecture. Shopify's 2025 retrospective on migrating its main app and POS to Fabric reported ~43% faster cold start, ~39% faster rendering, and ~25% memory reduction across millions of merchants. Discord, Coinbase (after their famous performance overhaul), Microsoft Office, Tesla, Walmart, and 99% of Instagram's UI all ship on RN today. The framework gap to Flutter, native Swift/Kotlin, and KMP has effectively closed for CRUD-heavy apps with adaptive layouts; **the gap that remains is in watchOS, advanced AVFoundation, complex BLE, and on-device ML, all of which require selective native modules anyway**.

For a workout-logging app, **Expo SDK 54 is the right target now** (last release supporting Legacy Arch fallback and the only one Clerk's passkey SDK formally supports as of April 2026). SDK 55 is fine but tighter. **Use Expo Development Builds, not Expo Go** — every premium feature you need (BLE, HealthKit, MMKV, Live Activities, RevenueCat) requires a custom dev client. Expo Router v6 is canonical; use the new (still `unstable-`) **Native Tabs** API for true UIKit/Compose tab bars with iOS 26 Liquid Glass — fall back to JS Tabs if a specific screen breaks.

The state-management debate is largely settled. **Keep Zustand + TanStack Query v5**, persist both to MMKV (synchronous, ~30× faster than AsyncStorage). Wire `onlineManager` to NetInfo for offline detection and use `PersistQueryClientProvider` with `resumePausedMutations` for queue-on-reconnect. Recoil was archived in January 2025; ignore it. **Legend State v3** is the only library worth a second look — it bundles state, persistence, and sync into one signals-based primitive, but its community is smaller and your proposed stack already has battle-tested coverage. **FlashList v2** (Shopify, 2025) is the list. The v2 rewrite eliminated the `estimatedItemSize` footgun and is already the engine behind every list in the Shopify mobile app.

For "**never lose a logged set**," combine three patterns: SQLite WAL mode with `busy_timeout=2000` and `synchronous=NORMAL`; optimistic UI via TanStack Query mutations or Legend State's sync engine; and a dedicated MMKV instance (`{id: 'active-workout'}`) for the in-flight set, flushed to SQLite on `appState === 'background'`. Keep the active-workout tab mounted (`unmountOnBlur: false`) so timers and BLE subscriptions don't churn during a 60-minute session.

## The local-first layer is your most consequential decision

PowerSync is the right choice and has no real competition for your shape of problem. **WatermelonDB has slowed to a release every 9 months and still requires you to build the entire sync server yourself** — multiple production teams (visible in the Drift discussion #2880 thread) have explicitly migrated *from* WatermelonDB *to* PowerSync once they hit conflict-resolution pain. **ElectricSQL's 2024 pivot left a long-polling, write-it-yourself system that one engineer in a March 2026 review called "radioactive waste" for greenfield apps**, and it has no first-class React Native SDK. **Rocicorp's Zero shipped 1.0 in early 2026** and is a beautiful piece of engineering, but it has no React Native client; it's web-only. **Triplit was acqui-hired into Supabase in October 2025** and is now community-maintained — Supabase is signaling first-party offline-first capability for the future, but it isn't shipped yet. **LiveStore** is fascinating for personal-data apps but its "one user = one SQLite instance" model breaks when you add gym-shared programs and friend leaderboards. **PowerSync remains the only mature, vendor-supported, React-Native-first sync engine for an offline-first SQLite app pulling from Postgres in 2026**.

Two PowerSync-specific recommendations matter. First, **adopt Sync Streams (beta but production-ready) on day one and skip the legacy Sync Rules YAML/buckets model**. Sync Streams supports JOINs, nested subqueries, TTL caching, and `auto_subscribe: true` for offline-first preload — a much better fit for partitioning data by user and gym tenant than the rigid bucket model. Second, **expect schema migrations to TRUNCATE before DROP** to avoid stale data and **expect column renames to retransmit entire tables**; design your schema with this in mind.

PowerSync pricing is reasonable. The Pro plan starts at $49/mo and includes 30 GB synced + 1k peak concurrent. Their own published worked example (chat-shaped data, denser than fitness logs) puts a 50k-install / 5k-DAU app at $51/mo Pro, a 1M-install app at ~$399/mo, and a 10M-install app at ~$6,000/mo. Workout logs are mostly insert-only, so conflicts are rare; expect to track below those numbers. Plan for **Team plan ($599/mo) at the point you need SOC 2/HIPAA-eligible BAA** — you don't today (HIPAA does not apply to a B2C fitness app — see security below).

## Supabase stays, but configure it like a senior team

Supabase is the right backend, but it has known production sharp edges. **Index every column referenced in an RLS policy** and **wrap `auth.uid()` in `(select auth.uid())`** — without these patterns RLS can be 100× slower (170ms vs 0.012ms in their published benchmarks). For Postgres connections from long-lived Node 22 workers, **use the direct connection on port 5432 with the IPv4 add-on, not Supavisor** — the 2025-2026 Supavisor + Vercel Fluid Compute connection-leak bug (where client connections grow unbounded over days, exhausting the pool) is documented across multiple GitHub discussions and has bitten Drizzle and Prisma users alike. Supavisor transaction mode (port 6543) remains the right choice for serverless edge handlers.

Supabase has had real outages: a multi-region April 2026 incident required manual project restarts, and the June 2025 GCP outage took down many Supabase projects alongside Firebase, Cursor, and Lovable. **Status-page history shows ~800 incidents over three years**; design your sync layer to buffer and retry. The Auth MAU pricing — **$0.00325/MAU above the included tier** — translates to **$3,250/mo per million users in auth costs alone**, which is the largest scaling surprise in the stack. This is the single strongest argument for fronting Supabase Auth with Clerk-issued JWTs (see authentication below).

**Drizzle ORM is the clear winner over Prisma in 2026.** Even after Prisma 7 (Nov 2025) killed its Rust query engine and shrunk to 1.6 MB gzipped, Drizzle is still ~130× smaller, has zero codegen step, native edge-runtime support without proxies, and — since PlanetScale acquired the Drizzle team in Q1 2026 — a real corporate backer. Drizzle even has an experimental client-side adapter for PowerSync's local SQLite, so you can use the same query layer end-to-end. Keep Drizzle.

**Change your worker hosting plan.** Supabase Edge Functions are Deno on a regional edge with cold starts; they cannot hold a persistent BullMQ Redis connection. **Run BullMQ workers on Fly.io (Hono on Bun, multi-region) or Railway (Node 22, simplest)** — both keep long-lived TCP open. Use Edge Functions only for webhook receivers (Stripe, RevenueCat, Mux, App Store Server Notifications V2). And **change Upstash Redis pay-as-you-go to Redis Cloud or Upstash Fixed plan**: BullMQ polls Redis constantly via BLPOP loops and scheduled-job timers, and Upstash's own documentation warns that PAYG pricing will produce "extra costs" against this access pattern. Redis Cloud's fixed pricing with persistent TCP is the right shape; DragonflyDB Cloud is a viable cost-optimized alternative once you cross 100k DAU.

## Authentication: Clerk in front of Supabase is the 2026 best practice

**Use Clerk as the identity provider, configured as a Supabase "Third-Party Auth" provider, with PowerSync verifying the same Clerk-issued JWT directly via Clerk's JWKS.** This single architecture solves five problems at once: native passkey login (Clerk has it; Supabase Auth still does not, four years after the GitHub issue opened), native Apple/Google sign-in via ASAuthorization and Android Credential Manager (Clerk's `useSignInWithGoogle()` shipped March 2026), an out-of-the-box **Organizations primitive** that maps cleanly onto your gym→trainers→members hierarchy, the new April 2025 first-class Supabase integration (the legacy JWT-template approach is deprecated), and one JWT verified everywhere — Supabase RLS, PowerSync, your Hono API, and any future edge functions.

**Clerk's pricing changed favorably in February 2026.** The free tier expanded to **50,000 monthly retained users** (up from 10k); above that, Pro is $25/mo + $0.02/MAU. At 10k MAU you pay nothing; 100k MAU is roughly $1,025/mo; 1M MAU is roughly $19,000/mo. Compare that to Supabase Auth's $3,250/mo per million MAU and you can see the trade-off: **Clerk gives you better DX, native components, passkeys, and B2B primitives at the cost of ~6× the auth bill at 1M MAU**.

The right approach is **start on Clerk, design every RLS policy to use `auth.jwt() ->> 'sub'` (which works identically for Clerk and Supabase JWTs), and keep Supabase Auth as your migration safety net** if Clerk pricing becomes hostile past 500k MAU. Both providers issue RS256/JWKS tokens; the cutover is a roughly two-week project, not a rebuild.

A few hard rules: **do not use Auth0** for a B2C fitness app — their November 2025 pricing hike took B2C Essentials to $70/mo for 500 MAU and overage to $0.07/MAU, which is wildly hostile. **Do not use WorkOS as primary auth for RN** — they have no native React Native SDK; their AuthKit is web-redirect only. **Stytch was acquired by Twilio in November 2025** and roadmap uncertainty makes it a bad multi-year bet. **Custom JWT** only makes sense if you have a Go/Rust auth team and a regulator-mandated reason to self-host.

For native sign-in: **`expo-apple-authentication` is required** (Apple's "Sign in with Apple" mandate is still enforced by App Review even after the January 2024 softening; the new January 2026 Korea-only S2S notification endpoint requirement is handled automatically by both Clerk and Supabase). **Use `@react-native-google-signin/google-signin` v20+ with Credential Manager**, not the deprecated `expo-auth-session` Google flow that broke in SDK 53 and which Expo maintainers have explicitly stopped supporting. **`expo-secure-store` with `keychainAccessible: AFTER_FIRST_UNLOCK_THIS_DEVICE_ONLY` is mandatory** for refresh tokens; access tokens should live in memory or SecureStore.

For **multi-tenant gym RBAC**: model each gym as a Clerk Organization, lifters who pay individually have no org, trainers and gym admins are members of one or more orgs with `org:trainer` or `org:admin` roles. The `org_id` and `org_role` claims arrive automatically in the JWT and your Supabase RLS policies read them via `auth.jwt() ->> 'org_id'`. PowerSync Sync Streams partition the same way — gym data by `org_id`, personal data by `sub`.

## Subscriptions: RevenueCat now, Superwall later, never raw StoreKit

**Day-one stack: RevenueCat alone**, on its new "Basic" free tier (genuinely free up to $10K MTR, includes the no-code Paywall Builder). The classic free-up-to-$2,500-MTR Pro tier remains for full features (Experiments, webhooks, integrations). Above $2,500 MTR, RevenueCat takes 1% of gross MTR — which works out to ~1.18% of net revenue under Apple's 15% Small Business Program rate. **At 100k users with a 5% paid rate and $7.83 blended ARPU, RC costs ~$391/mo on $33,278/mo net revenue — about 0.07% of run-rate**. This is cheap insurance against Apple's annual StoreKit changes, Google Play Billing v8 → v9 migration (expected mid-2026), JWS validation, family-sharing edge cases, and EU DMA accounting; **going raw StoreKit 2 + Play Billing burns roughly one engineer-month per year in maintenance** and is not justifiable until you cross $5M ARR.

**Add Superwall when paywall iteration becomes a bottleneck — typically around $25K MTR.** Superwall's pricing repriced in 2025 to free up to $10K MAR (revenue *attributed to* a Superwall paywall, not total revenue) and 1% MAR after, which makes it stackable on top of RevenueCat at minimal incremental cost. The pattern: **RevenueCat owns purchases and entitlements; Superwall owns paywall UI, behavioral triggers, and experiments**. RevenueCat's dashboard has a built-in Superwall integration toggle that forwards purchase events. Adapty is a credible alternative to RevenueCat with a slightly more flexible paywall builder, but the migration cost off RevenueCat later is the deciding factor — pick one, stick with it.

**Server-side entitlement architecture**: trust the RevenueCat SDK for in-session UX (cached `customerInfo.entitlements`, sub-millisecond), but maintain a Supabase `entitlements` table as the durable source of truth, populated by RC webhooks from a Supabase Edge Function. Use `event.app_user_id` (not `body.app_user_id`), enforce idempotency via `last_event_id`, return 200 even on duplicates to prevent retry storms, and run a daily reconciliation cron that calls RC's REST API for users whose `expires_at < now()` to catch missed webhooks. **Always call `Purchases.logIn(supabaseUserId)` before any purchase attempt** — otherwise RC creates an `$RCAnonymousID` that won't match your users.

**On the EU and US tax debate**: the EU DMA's January 2026 Core Technology Commission replaced the old per-install CTF with a 5% revenue commission on top of 5%/13% Store Services and 2% Initial Acquisition Fee. **Best-case external payments in the EU work out to ~13%, worst-case ~23%** — the math doesn't beat Apple's 15% Small Business rate for fitness apps until you cross $1M ARR. In the US, the Epic v Apple injunction has Apple currently charging zero commission on external link purchases (as of May 5, 2026, Apple filed an emergency stay application with the Supreme Court). **Build the integration feature-flagged**; expect commissions to return at 5-15% once the district court sets a "reasonable" rate.

**Fitness-specific paywall benchmarks** (RevenueCat and Adapty 2026 State of Subscription Apps): Health & Fitness has a **5× higher trial-to-paid conversion on hard paywalls vs freemium** (10.7% vs 2.1% at D35). 80-82% of trial starts happen on Day 0, so the onboarding paywall is non-negotiable. **7-day free trials hit the conversion sweet spot** (37.4% trial-to-paid for 5-9 day trials vs 25.5% for ≤4 days). Annual:monthly mix should target ~70:30 with annual at $59.99-$79.99 and monthly at $9.99-$14.99. Hevy added a $74.99 lifetime tier in 2026; consider mirroring at the 18-month mark.

## Security: HIPAA does not apply, MHMDA does, and stop pinning certs

The single most common security mistake on fitness apps is **claiming HIPAA compliance you don't have and don't need**. HHS itself has stated HIPAA "does not apply and was never intended to apply to general health and wellness applications." MyFitnessPal, Fitbit, Wahoo, and Runkeeper are all explicitly out-of-scope; your fitness app is too. HIPAA only triggers when you act as a Business Associate to a Covered Entity under a signed BAA, or integrate clinically (telehealth, EHR push). Don't market HIPAA, but **do design "as if"** so a future enterprise wellness deal with a self-funded health plan is cheap to enable.

The actual US compliance exposure is **Washington's My Health My Data Act (MHMDA)**, effective March 31, 2024. MHMDA's "consumer health data" definition is intentionally broad — body weight and workout logs are explicitly in scope per Stoel Rives' analysis. You must publish a **separate Consumer Health Data Privacy Policy** as a distinct link from your main policy, obtain **opt-in consent before collection** (separate consent for sharing), honor deletion requests with propagation to processors, and avoid geofencing health facilities. There's a private right of action under WA's Consumer Protection Act, which makes this the highest-leverage state-law risk on US launch. **CCPA's January 2026 amendments** (mandatory risk assessments for sensitive personal information, GPC honoring, ADMT disclosures) raise the California bar — Healthline's $1.55M July 2025 settlement was the AG's largest CCPA fine to date and was specifically about a health-adjacent app. The **FTC Health Breach Notification Rule** also applies to non-HIPAA health apps with a 60-day notification window.

For **GDPR**, the right-to-erasure (Art. 17) is now an EDPB enforcement priority per their February 2026 CEF report. Build in-app account deletion that propagates to Supabase, PowerSync, Sentry, PostHog, and RevenueCat within 30 days, with documented backup retention. In-app data export (Art. 20) in JSON is mandatory. DPAs (Art. 28) with every processor, SCCs/DPF for US transfers, and a documented DPIA for "large scale" health processing are non-negotiable.

On the technical side, **most fitness apps over-invest in security theater**. **Skip certificate pinning** — its main value is defending against rogue CAs and corporate MITM, the cost is bricked apps when a cert rotation is botched, and React Native's official docs explicitly warn about expiry. **Skip JS obfuscation** beyond Hermes + R8 — Hermes bytecode is reversible by `hermes-dec` (P1 Security) and `hbctool` in minutes; OWASP MASTG documents the tooling. Treating Hermes as a security boundary is a marketing claim, not engineering reality. **Skip blocking rooted/jailbroken devices** — `jail-monkey` is one Frida script away from defeat; a soft warning is fine, a hard block angers power users.

What to actually do: **enable Hermes + R8 (default)**, **enforce TLS 1.2+ via ATS and Android Network Security Config** (`usesCleartextTraffic=false`), **store refresh tokens in `expo-secure-store`** with `requireAuthentication: true` for destructive actions, **rotate refresh tokens on every use** with `jti` reuse detection that revokes the entire token family, **use Universal Links and App Links** (never custom schemes) with server-side validation of every payload, and **gate sensitive backend mutations on `@expo/app-integrity`** (Apple App Attest + Google Play Integrity Standard). For the on-device SQLite layer, **OS-level encryption is sufficient for workout logs**; layer **SQLCipher via `@op-engineering/op-sqlite`** only when HealthKit/Health Connect data lands on device in Phase 2. The OWASP Mobile Top 10 2024 ranks **M1 Improper Credential Usage and M3 Insecure Authentication** above all else — your 2026 security budget should go to credential hygiene, supply-chain scanning (npm audit, Dependabot, SBOM), and PII scrubbing in Sentry/PostHog rather than reverse-engineering theater.

## Supporting infrastructure: PostHog, Postmark, Sentry, Mux, Live Activities

**Replace separate analytics, feature-flag, replay, and error vendors with PostHog.** The 2026 free tier is 1M events/month plus 5,000 web replays plus 2,500 mobile recordings plus unlimited feature flags. The React Native SDK is production-grade with native peer deps and an Expo plugin; session replay v3.2+ shipped via `posthog-react-native-session-replay` with privacy-safe wireframe mode by default. Bundling analytics + flags + replay + experiments + errors into one tool dwarfs the depth advantage of Amplitude/Mixpanel/LaunchDarkly at your scale, and the unified DX (ship behind flag → watch replays of users on the new variant → check funnel impact → catch errors) is genuinely faster. Add **RevenueCat for subscription analytics** (different domain, doesn't overlap). Skip Statsig post-OpenAI acquisition uncertainty.

**Use Sentry for crash reporting and performance monitoring.** Hermes source map handling is mature, New Architecture is fully supported, UI Profiling reached GA on Android (SDK 7.9+) and iOS (7.12+), and Session Replay for RN is available. Pricing is usage-based and starts free at 5K errors/month. **Embrace** is the tier-2 consideration when crash-free rate becomes a board metric — its 100% session capture and native-layer ANR/OOM visibility is genuinely better, and its OSS SDK is OpenTelemetry-compliant — but defer until you need it. Reject Bugsnag (Sentry has eaten its lunch in RN) and Datadog RUM (enterprise economics).

**Email: Postmark, not Resend or SendGrid.** Postmark's measured 98.7% inbox placement vs SendGrid's 95.3% (and 1-second delivery vs SendGrid's 64s) is a real retention dollar advantage when workout reminders are a core engagement loop. SendGrid killed its free tier in May 2025; Postmark starts at $15/mo for 10k messages. Resend is excellent DX and fine for transactional, but for premium positioning the deliverability gap matters. **Layer Customer.io at 10k+ users** for behavioral journeys (workout streaks, win-back drips, weekly digests). Skip OneSignal, Braze, and Iterable for V1.

**Push notifications: Expo Notifications for V1 — free, fast (p50 42ms vs OneSignal's 222ms), and the right default.** Layer Customer.io when you need segmented behavioral campaigns.

**Live Activities are a category-defining feature for premium fitness apps in 2026; ship them in V1.** `expo-live-activity` (Software Mansion) supports push-driven start/update/stop, Dynamic Island layouts, and timer fields anchored to server time so SwiftUI renders the countdown locally without per-second JS updates. For richer activities (live HR, set count, RPE), graduate to `expo-apple-targets` (Evan Bacon) for full Swift widget extension control. The new Expo `expo-widgets` library is the unified path going forward.

**Video: keep Mux unless cost dominates.** At 100k MAU watching ~30 min/month of demos, Mux runs ~$3,300/mo (after the 100k free min/mo allowance and auto cold-storage tiering). bunny.net Stream is roughly 100× cheaper at the same scale ($25/mo) but has weaker analytics, fewer DRM options, and a lighter dashboard — a credible move if your premium positioning shifts to cost-conscious. Cloudflare Stream doesn't reduce price for sub-1080p (where exercise demos live), so it's the wrong middle ground. **Use `expo-video` with `expo-video-cache`** — the latter spins up a localhost proxy on iOS to cache HLS segments (AVPlayer can't natively cache HLS; this is the canonical workaround).

**iOS background sync is "eventually consistent" — design for it.** `BGAppRefreshTask` is a 30-second hint, not a schedule; iOS routinely delays it by hours. The new iOS 26 `BGContinuedProcessingTask` (WWDC 2025) is the right choice for user-initiated long-running tasks like data export. **Silent push is your most reliable wake mechanism** for cross-device sync, but iOS throttles aggressively for force-quit apps. Reconcile state on every foreground; never assume background tasks ran.

**Apple Watch must be native SwiftUI.** React Native does not run on watchOS — there is no JS runtime, no path, no shortcut. Use `react-native-watch-connectivity` to bridge state between the RN iPhone app and a native SwiftUI Watch app you build separately in Xcode. Use HealthKit as the source of truth where possible (auto-syncs phone↔watch via iCloud); reserve WatchConnectivity for app-specific state (current workout template, set count). Build the Watch app as **independent** (post-watchOS 6 model) so it works without the phone in range — fitness users leave their phone at home. This is Phase 2/3 work; hire a Swift contractor.

## Where this stack actually fails

Three blunt realities. **First, RN is not native parity for everything**, and Shopify's own retrospective explicitly says "100% React Native should be an anti-goal." Native is still better for Apple Watch, complications, App Intents, Siri Shortcuts, advanced AVFoundation, complex BLE state restoration, on-device ML, and long-running background work. Plan for selective native modules from day one — they're not failure, they're the architecture.

**Second, the RN cold-start floor on mid-range Android is still ~1,800ms** even with Hermes V1 and the New Architecture. Coinbase's documented performance crisis (JS FPS dropping to single digits on a hot path, requiring a custom Babel plugin to wrap every JSX element) is the canonical reminder that you must measure your performance budget continuously. Wire up Sentry's UI Profiling and Time-to-Initial-Display from day one.

**Third, PowerSync's case-study list skews enterprise/industrial.** No major consumer fitness app has publicly committed to PowerSync; you'd be the most prominent reference customer. The Sync Streams beta is production-ready but new, the operations-history can balloon without compaction (and trigger billing surprises), the WASM-SQLite path on web has measurable latency, and PowerSync explicitly does not sync attachments — **handle exercise demo videos and progress photos via Mux and Supabase Storage, not PowerSync**. The single biggest documented PowerSync regret pattern is teams running it without tracking operations-history bloat; instrument it from week one.

## Conclusion: the four moves that actually change outcomes

Your stack is fundamentally right. The four changes that will compound the most over the next 18 months are: **(1) replace Upstash PAYG with Redis Cloud or Upstash Fixed and host BullMQ workers on Fly.io/Railway** — this fixes a billing landmine and a runtime-shape mismatch in one move; **(2) put Clerk in front of Supabase as your IdP from day one** so you ship native passkeys, native Apple/Google sign-in, and gym-org RBAC without rebuilding when you outgrow Supabase Auth's MAU pricing; **(3) commit to RevenueCat from the first build** and design your Supabase entitlement table around their webhook schema so layering Superwall in 12-18 months is a configuration change, not a migration; **(4) build the Apple Watch app native from the start of Phase 2** rather than attempting an RN-on-watchOS detour that does not exist.

The rest is execution discipline: **Sync Streams over legacy Sync Rules**, **direct Postgres connections from long-lived workers (not Supavisor)**, **indexed RLS policies** with `(select auth.uid())`, **Live Activities in V1** as a premium-tier moat, **PostHog + Sentry + Postmark** as a unified observability and engagement layer, and **MHMDA-grade privacy hygiene** (separate consumer health data privacy policy, opt-in consent, 30-day deletion propagation) rather than HIPAA theater. Skip the security panic (no cert pinning, no JS obfuscation, no root-blocking) and redirect that budget to App Attest/Play Integrity attestation on sensitive endpoints and dependency-supply-chain scanning, which is where the real 2026 attack surface lives.

If you ship the stack above, you will have built a closer technical analog to Hevy (RN + Node + Postgres) with a stronger offline-first guarantee than Hevy currently advertises, a better paywall iteration loop than Strong, native Watch parity with Fitbod and Future, and a B2B-tenant primitive (Clerk Organizations) that none of the consumer fitness incumbents currently have. That last point is the only one that compounds into a category-changing wedge — the gym operator persona is where the rest of the market hasn't built.
